# mdkg.dev Conversion Paths Strategy

## 1. Purpose

This document defines the conversion strategy for mdkg.dev.

The site should route different visitor types without diluting the core developer path. The primary goal is not immediate revenue. The launch should build trust, usage, GitHub/social proof, docs engagement, and a bridge to Nicholas Reames and ochatr.ai.

## 2. Conversion thesis

> mdkg.dev should turn attention into developer action: understand the promise, try the CLI, star/follow the project, read docs, and optionally join the larger ochatr.ai orbit.

The primary conversion is not “join a waitlist.” It is:

1. Understand Markdown Knowledge Graph.
2. Try it or inspect GitHub.
3. Trust Nicholas as a serious AI-native software builder.
4. Follow, star, install, or join the next layer.

## 6. Conference-specific conversion strategy

The conference goal is conversations, not mass broadcast.

### In-person pitch

> I’m building Markdown Knowledge Graph, a git-native project memory layer for AI coding agents. It turns Markdown and frontmatter in a repo into deterministic context packs, goals, checkpoints, skills, and handoffs.

### Best opening question

> How are you keeping AI coding agents aligned with project context across sessions?

### When someone is interested

1. Open personal site QR/NFC.
2. Point to Markdown Knowledge Graph.
3. Offer GitHub/docs link.
4. Ask whether to connect on X, LinkedIn, or GitHub.
5. Mention ochatr.ai only if they ask about long-term vision.

### Conference landing behavior

Personal site should be the tap target because it routes to everything. mdkg.dev should be the product proof.

## 11. Risk management

### Risk: mdkg.dev becomes too many CTAs

Mitigation:

- Homepage CTA hierarchy: install/GitHub/docs first, socials/ochatr later.

### Risk: ochatr.ai dilutes mdkg launch

Mitigation:

- Keep ochatr as tertiary bridge.
- mdkg must stand alone.

### Risk: advanced features confuse first-time users

Mitigation:

- Advanced alpha preview only on homepage.
- Deep advanced docs in GitBook.

### Risk: personal site becomes a distraction

Mitigation:

- Keep it as a simple mobile link hub.
- No blog/CMS before launch.

### Risk: social posts have no conversion path

Mitigation:

- Every launch post should point to mdkg.dev or GitHub/docs.

## 13. Summary

The conversion strategy should preserve the main launch message: Markdown Knowledge Graph is useful now. mdkg.dev should convert attention into technical trust first, then into social following and ochatr.ai waitlist interest second.
