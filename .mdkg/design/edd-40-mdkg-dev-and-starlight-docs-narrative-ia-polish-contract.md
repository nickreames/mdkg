# Narrative Contract

Use one public model everywhere: `Plan -> Work -> Evidence`.

Definition:
- Plan the goal.
- Execute one bounded work node.
- Record evidence.
- Validate before moving on.

Avoid competing public phrases such as golden loop, repo memory loop, or generic operating loop.

# Marketing Site Contract

- Hero names the audience, problem, and differentiator quickly.
- First-run setup is visually separate from ongoing work-loop commands.
- Homepage includes a simple accessible Plan -> Work -> Evidence diagram.
- Low-dependency and local-first posture are visible early.
- Work-node summary explains familiar SDLC shapes without metadata overload.
- Advanced alpha content is short and clearly secondary.
- `/docs` contains no Starlight/GitBook/future-host implementation commentary.

# Docs Contract

- Docs homepage routes users to install, quickstart, core concepts, and safety.
- Work Node Types explains lifecycle and goal routing.
- Reference Types explains `scope_refs`, `context_refs`, and `evidence_refs` with realistic frontmatter.
- Repository Layout explains committed, generated, optional, mirrored, commit, and do-not-commit paths.
- Agent Workflow includes copy-pasteable agent instructions.
- Packs and Handoffs explains when to use each and what not to put into them.
- CLI Reference is user-first; Command Contract is maintainer/integration-facing.
- Roadmap is product-oriented.

# Starlight Navigation

- Short/index pages should not show useless right-side TOCs.
- Real concept/guide pages should have useful H2/H3 anchors.
- Claims Evidence Matrix is not in public navigation.
- Sidebar labels should make beginner, advanced, reference, and project sections clear.
