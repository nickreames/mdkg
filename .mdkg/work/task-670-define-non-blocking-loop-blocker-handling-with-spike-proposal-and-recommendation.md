# Overview

Define loop blocker behavior that keeps progress moving. A loop should not mark
itself blocked easily when one branch hits uncertainty; it should create or
request structured investigation and continue other useful scoped work where
possible.

# Acceptance Criteria

- Blocked branch handling requires or recommends a spike with source-grounded
  and, when needed, web-grounded options.
- A proposal with at least three viable paths is required for non-trivial
  blockers.
- The proposal records a recommended path forward.
- Affected goals or subgoals receive explicit blocker evidence.
- The loop then searches for other scoped work that advances the definition of
  done.
- True loop-level blocking requires repeated or global blockers, not one early
  branch failure.

# Blocker-Continuation Algorithm

When a loop branch hits a blocker, the loop should not immediately mark the
whole loop blocked. It should:

1. Identify the blocked branch and the affected goals/subgoals.
2. Record blocker evidence on the affected goal, subgoal, task, or loop branch.
3. Create or request a spike when uncertainty is material and source-grounded
   investigation is required.
4. If external current facts are required, the spike should explicitly ask the
   executing agent/harness to use web search plus source-code grounding.
5. Create or request a proposal for non-trivial blockers.
6. Require at least three viable paths in that proposal.
7. Record a recommended path and why it is preferred.
8. Decide whether the blocked branch is deferred, waiting on approval, waiting
   on external state, or ready for the recommended path.
9. Search the loop scope for other useful work that advances the definition of
   done.
10. Continue with that work when safety and ownership rules allow.

# Whole-Loop Blocking Threshold

A loop should only treat itself as blocked when blockers are repeated, global,
or prevent meaningful progress across the remaining scope. A single branch
blocker should produce structured investigation and continue-progress behavior.

# Implementation Notes For Future Tests

- Test fixtures should prove a blocked branch can create/link a spike and
  proposal while the loop remains active or planned.
- Proposal validation should require at least three options and a recommendation
  when generated by blocker-continuation behavior.
- The runtime or agent owns actual research, web browsing, tool execution, and
  approval requests. mdkg owns the state model and validation contract.
