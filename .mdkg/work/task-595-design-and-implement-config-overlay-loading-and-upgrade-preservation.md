# Overview

Implement the config-overlay and upgrade-preservation behavior needed for
companies to customize mdkg standards after init while keeping the mdkg kernel
upgradable.

# Acceptance Criteria

- `.mdkg/config.json` can represent organization standards/overlays without
  requiring an init-time profile.
- `mdkg upgrade` reports overlay-aware planned changes.
- `mdkg upgrade --apply` preserves local overlays, custom core docs, and custom
  mirror targets unless the operator explicitly accepts migration.
- Default initialized workspaces continue to validate without custom overlays.

# Files Affected

- `src/core/config.ts`
- `src/commands/upgrade.ts`
- `assets/init/config.json`
- focused tests and docs generated by the implementation task

# Implementation Notes

- Do not introduce a forkable starter repo as the primary path.
- Keep schema additions conservative and backward compatible.
- Upgrade receipts should explain preserve/update/skip decisions.

# Test Plan

- Unit tests for config parsing and migration defaults.
- Temp-repo smoke: init, customize overlays, run upgrade dry-run, run upgrade
  apply, verify custom policy remains.
